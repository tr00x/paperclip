<task type="auto">
  <name>Task 1: Deploy Twenty CRM via Docker Compose</name>
  <files>twenty-crm/docker-compose.yml, twenty-crm/.env</files>
  <read_first>
    - https://raw.githubusercontent.com/twentyhq/twenty/main/packages/twenty-docker/docker-compose.yml
    - https://raw.githubusercontent.com/twentyhq/twenty/main/packages/twenty-docker/.env.example
  </read_first>
  <action>
    1. Create directory `twenty-crm/` in the project root.

    2. Download the official Twenty docker-compose.yml:
       ```
       curl -sL https://raw.githubusercontent.com/twentyhq/twenty/main/packages/twenty-docker/docker-compose.yml -o twenty-crm/docker-compose.yml
       ```

    3. Create `twenty-crm/.env` with these values:
       - TAG=latest
       - SERVER_URL=http://localhost:3000
       - PG_DATABASE_PASSWORD=(generate a strong password with `openssl rand -hex 16`)
       - APP_SECRET=(generate with `openssl rand -base64 32`)
       - STORAGE_TYPE=local

    4. Start the stack:
       ```
       cd twenty-crm && docker compose up -d
       ```

    5. Wait for healthy status:
       ```
       docker compose ps
       ```
       All 4 services (server, worker, db, redis) must show "running" or "healthy".
       The server health check hits /healthz every 5 seconds — wait up to 2 minutes for it.

    6. Verify Twenty is accessible:
       ```
       curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/healthz
       ```
       Expected: 200
  </action>
  <verify>
    <automated>cd /Users/timur/paperclip/twenty-crm && docker compose ps --format json | grep -q '"Status"' && curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/healthz</automated>
  </verify>
  <acceptance_criteria>
    - twenty-crm/docker-compose.yml exists and contains "twentycrm/twenty" image reference
    - twenty-crm/.env exists and contains APP_SECRET, PG_DATABASE_PASSWORD, SERVER_URL values
    - `docker compose ps` in twenty-crm/ shows 4 services running (server, worker, db, redis)
    - `curl http://localhost:3000/healthz` returns HTTP 200
  </acceptance_criteria>
  <done>Twenty CRM is running locally at http://localhost:3000 with all 4 Docker services healthy.</done>
</task>

<task type="auto">
  <name>Task 3: Create 4 pipelines and get API key via Twenty API</name>
  <files>twenty-crm/.env</files>
  <read_first>
    - twenty-crm/.env
  </read_first>
  <action>
    After user completes workspace setup, use the Twenty GraphQL API to configure pipelines and get an API key.

    1. First, get the admin API key. Navigate to Settings > Accounts > API Keys in Twenty UI, or use the Twenty CLI. The user may need to provide the API key from the UI. If the API key was provided by the user, proceed with it.

    2. Using the Twenty GraphQL API at http://localhost:3000/api, create 4 pipeline stages (Twenty uses "pipeline steps" via the Opportunity object's stage field). In Twenty CRM, pipelines are managed through the "Opportunities" object with custom pipeline stages.

       Create/verify these pipeline configurations exist:
       - **Leads** pipeline — for tracking prospective clients found by Hunter
       - **Tenders** pipeline — for government contract opportunities from Gov Scout
       - **Clients** pipeline — for active client accounts
       - **Invoices** pipeline — for financial tracking by Finance Tracker

       Note: Twenty CRM may use custom objects or the built-in Opportunity stages. Check what's available via the API first:
       ```
       curl -s http://localhost:3000/api -H "Authorization: Bearer API_KEY" \
         -H "Content-Type: application/json" \
         -d '{"query": "{ objects { edges { node { nameSingular namePlural } } } }"}'
       ```
       Then create custom objects or configure opportunity stages as appropriate.

    3. Generate a dedicated API key for the MCP server (Phase 2):
       - Use Settings > API & Integrations in the Twenty UI, or the API
       - Save the API key

    4. Append the API key to `twenty-crm/.env`:
       ```
       TWENTY_API_KEY=<generated-key>
       ```
       This will be used by the twenty-mcp server in Phase 2.

    5. Verify API access works:
       ```
       curl -s http://localhost:3000/api -H "Authorization: Bearer TWENTY_API_KEY" \
         -H "Content-Type: application/json" \
         -d '{"query": "{ currentWorkspace { displayName } }"}'
       ```
       Expected: returns workspace name "AmriTech HQ"
  </action>
  <verify>
    <automated>grep -q "TWENTY_API_KEY=" /Users/timur/paperclip/twenty-crm/.env && curl -s http://localhost:3000/api -H "Authorization: Bearer $(grep TWENTY_API_KEY /Users/timur/paperclip/twenty-crm/.env | cut -d= -f2)" -H "Content-Type: application/json" -d '{"query": "{ currentWorkspace { displayName } }"}' | grep -q "AmriTech"</automated>
  </verify>
  <acceptance_criteria>
    - 4 pipelines (Leads, Tenders, Clients, Invoices) are visible in Twenty CRM
    - TWENTY_API_KEY is saved in twenty-crm/.env
    - GraphQL query to /api with the API key returns the workspace name
    - API key can list objects via the metadata API
  </acceptance_criteria>
  <done>Twenty CRM has 4 pipelines configured and a working API key saved for Phase 2 MCP integration.</done>
</task>
