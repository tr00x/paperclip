# Система саморазвития агентов

## Принцип

Агенты НЕ меняют свои файлы напрямую. Они предлагают изменения через `[IMPROVEMENT]` задачи. IT Chef или Tim ревьюит и накатывает.

Это как Pull Request — агенты растут, но под контролем.

## Что агент может улучшить

### 1. Свой SOUL.md
- Новый паттерн в работе: "Dental ниша отвечает на compliance angle лучше чем на cost angle"
- Новое правило из опыта: "Не писать email в пятницу вечером — 0% open rate"
- Уточнение к существующему правилу

### 2. Свой HEARTBEAT.md
- Новый шаг в процессе: "Перед email проверять website uptime — если сайт мёртв, не слать"
- Оптимизация порядка шагов
- Новый query для CRM

### 3. Свои скиллы
- Предложить новый skill для установки
- Предложить изменение к существующему skill

### 4. Memory / Known Issues
- ✅ ЭТО можно делать САМОСТОЯТЕЛЬНО — записывать факты, паттерны, lessons learned в свою память. Не требует ревью.

## Формат [IMPROVEMENT] задачи

Создай задачу в Paperclip:

```
Title: [IMPROVEMENT] {Agent Name}: {краткое описание}
Assignee: IT Chef
Priority: low

Description:
## Что предлагаю изменить
Файл: {путь к файлу}
Секция: {какая секция}

## Текущее поведение
{как сейчас}

## Предлагаемое изменение
{что хочу поменять — конкретный текст}

## Почему
{данные, паттерн, или инцидент который привёл к этому выводу}

## Ожидаемый результат
{что улучшится}

## Данные (если есть)
{например: "5 из 5 dental leads ответили на compliance angle, 0 из 5 на cost angle"}
```

## IT Chef Review Process

IT Chef получает [IMPROVEMENT] задачу и:

1. **Проверяет данные** — агент не выдумал? Есть реальные примеры?
2. **Проверяет конфликты** — не сломает ли это другие агенты или pipeline?
3. **Проверяет безопасность** — не убирает ли критичные правила (BCC, CRM update, approval gates)?
4. **Решение:**
   - ✅ **Approve** → IT Chef вносит изменение, рапортует в TG
   - ❓ **Escalate** → Передаёт Tim'у если сложное или рискованное
   - ❌ **Reject** → Объясняет почему, комментарий в задаче

### Что IT Chef НЕ одобряет (только Tim):
- Удаление BCC правила
- Изменение approval gates (кто что одобряет)
- Изменение escalation каскада
- Добавление нового агента
- Изменение CRM schema
- Изменения в инфраструктуре (watchdog, docker, webhook)

### Что IT Chef может одобрить сам:
- Новые паттерны/правила для конкретного агента
- Уточнения к существующим инструкциям
- Новые query примеры для CRM
- Оптимизация порядка шагов
- Установка read-only скиллов (документация, best practices)

## Формат одобрения

IT Chef в TG:
```
✅ [IMPROVEMENT] одобрен:
Агент: {name}
Изменение: {краткое описание}
Файл: {path}
Применено: {дата}
```

## Скиллы — установка новых

Агент может попросить установить skill:

```
Title: [SKILL-REQUEST] {Agent Name}: нужен skill {название}
Assignee: IT Chef

Description:
## Какой skill нужен
{название и описание}

## Зачем
{конкретная ситуация где он нужен}

## Где найти
{skills.sh URL или название из маркетплейса}
```

IT Chef проверяет что skill безопасный (read-only, не меняет файлы) и устанавливает.

## Что агент может делать САМОСТОЯТЕЛЬНО (без [IMPROVEMENT])

1. **Записывать в свою память** — факты, паттерны, lessons learned
2. **Обновлять known-issues** (только IT Chef)
3. **Предлагать идеи в TG** (формат 💡) — это не изменение, это предложение
4. **Адаптировать тон email'ов** в рамках существующих правил
5. **Менять приоритеты** в рамках своих полномочий

## Метрики саморазвития

CEO включает в weekly report:
```
🧠 Саморазвитие:
• [IMPROVEMENT] предложено: {N}
• Одобрено: {N}
• Отклонено: {N}
• Лучшее: {описание}
```
